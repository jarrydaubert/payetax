# Backlog

Only open TODO items live here.

## Rules

- No status updates, progress notes, or completed work.
- Every item must include a concrete next step and a clear, testable Definition of Done.
- If an item changes code or behavior, its default DoD expectation is automated regression protection at the right layer.
- Use manual validation only when automation is genuinely the wrong layer; capture that validation in the PR or linked Linear issue, not in standalone evidence docs.
- When an item is complete, remove it from this file.

## Backlog Mantra

- Open TODO items only.
- Clear, testable DoD only.
- Code and behavior changes must say how they stay fixed with automated coverage at the right layer wherever possible.
- Tests should catch the real bug or user-visible regression, not just assert implementation details.

## Definition Of Ready

A backlog item is ready to execute when all conditions below are true:

1. `Problem clear`: the row names the bug, gap, or user/business outcome.
2. `Next step concrete`: the next action is small enough to start without rediscovery.
3. `DoD testable`: completion can be proven by tests, validation commands, production checks, or a linked Linear closure.
4. `Risk named`: tax, privacy, SEO, performance, accessibility, or UX risk is obvious from the row.

## Global DoD Contract

A backlog item is only complete when all conditions below are true:

1. `Outcome delivered`: the row's scope is implemented exactly as written.
2. `Regression protection added`: code and behavior changes add or update automated coverage at the right layer wherever possible.
3. `Validation passed`: the appropriate repo gates or targeted checks pass.
4. `Linear closed or updated`: linked production issues are closed when fixed, or updated with the reason if no code change is needed.
5. `Backlog hygiene`: the completed item is removed from this file in the same change set.

If any condition is missing, the item remains open.
