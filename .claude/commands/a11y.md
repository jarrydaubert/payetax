## WCAG 2.2 AA Checklist

### 1. Perceivable

#### 1.1 Text Alternatives
- [ ] All images have meaningful `alt` text
- [ ] Decorative images use `alt=""` or `aria-hidden="true"`
- [ ] Complex images have extended descriptions
- [ ] Icons have accessible labels

#### 1.2 Time-based Media
- [ ] Videos have captions (if applicable)
- [ ] Audio has transcripts (if applicable)

#### 1.3 Adaptable
- [ ] Semantic HTML used (`<header>`, `<main>`, `<nav>`, `<section>`)
- [ ] Headings in logical order (h1 → h2 → h3)
- [ ] Lists use `<ul>`, `<ol>`, `<dl>` appropriately
- [ ] Tables have headers and captions
- [ ] Form inputs have associated labels

#### 1.4 Distinguishable
- [ ] Color contrast ratio ≥4.5:1 for normal text
- [ ] Color contrast ratio ≥3:1 for large text (18pt+)
- [ ] Color contrast ratio ≥3:1 for UI components
- [ ] Information not conveyed by color alone
- [ ] Text resizable to 200% without loss
- [ ] No horizontal scrolling at 320px width

### 2. Operable

#### 2.1 Keyboard Accessible
- [ ] All functionality available via keyboard
- [ ] No keyboard traps
- [ ] Logical tab order (follows visual order)
- [ ] Skip link to main content
- [ ] Focus visible on all interactive elements
- [ ] Custom components keyboard operable

#### 2.2 Enough Time
- [ ] No time limits (or adjustable)
- [ ] Auto-updating content can be paused

#### 2.3 Seizures
- [ ] No content flashes >3 times per second

#### 2.4 Navigable
- [ ] Page has descriptive `<title>`
- [ ] Focus order matches visual order
- [ ] Link text is descriptive (not "click here")
- [ ] Multiple ways to find pages (nav, search, sitemap)
- [ ] Headings describe content
- [ ] Focus indicator visible (2px+ outline)

#### 2.5 Input Modalities
- [ ] Touch targets ≥44x44px
- [ ] Gestures have alternatives
- [ ] Motion activation can be disabled

### 3. Understandable

#### 3.1 Readable
- [ ] Page language specified (`<html lang="en-GB">`)
- [ ] Abbreviations explained on first use

#### 3.2 Predictable
- [ ] Navigation consistent across pages
- [ ] Components behave consistently
- [ ] No unexpected context changes on focus

#### 3.3 Input Assistance
- [ ] Errors clearly identified
- [ ] Error messages descriptive
- [ ] Labels or instructions for inputs
- [ ] Error prevention for important actions

### 4. Robust

#### 4.1 Compatible
- [ ] Valid HTML (no parsing errors)
- [ ] ARIA used correctly
- [ ] Custom components have proper roles
- [ ] Status messages use `aria-live`

## ARIA Best Practices

### Landmarks
```html
<header role="banner">
<nav role="navigation" aria-label="Main">
<main role="main">
<footer role="contentinfo">
```

### Interactive Components
```html
<!-- Buttons -->
<button aria-label="Close dialog" aria-pressed="false">

<!-- Expandable -->
<button aria-expanded="false" aria-controls="menu">
<div id="menu" aria-hidden="true">

<!-- Live regions -->
<div aria-live="polite" aria-atomic="true">
```

### Form Patterns
```html
<!-- Required fields -->
<input aria-required="true" aria-describedby="hint">
<span id="hint">Format: £XX,XXX</span>

<!-- Errors -->
<input aria-invalid="true" aria-errormessage="error1">
<span id="error1" role="alert">Please enter a valid salary</span>
```
